以下术语和定义适用于本标准。

ISO和IEC通过以下地址维护标准化的术语数据库：

其他术语的定义见标准《ISO/IEC/IEEE 24765 系统和软件工程——词汇》：<www.computer.org/sevocab>

<!--ISO/IEC/IEEE 12207:2017# 新增术语-->

### 3.1.6 架构 architecture
<!--ISO/IEC/IEEE 12207:2017# 新增术语><GB/T 22032:征求意见稿# 4.1.5 架构 architecture-->

[来源： ISO/IEC/IEEE 42010:2011]

### 3.1.7 架构框架 architecture framework
<!--ISO/IEC/IEEE 12207:2017# 新增术语><GB/T 22032:征求意见稿# 4.1.6 架构框架 architecture framework-->

<!--GB/T 22032:征求意见稿# 在特定应用领域、利益相关方团体内，描述架构的惯例、原则和实践。-->
示例1：
通用企业参考体系架构和方法（GERAM）[ISO 15704]是一种体系架构框架。
<!--GB/T 22032:征求意见稿# 示例1：一般企业参考架构、方法论是架构框架。-->
示例2：
开放分布式处理参考模型（RM-ODP）[ISO / IEC 10746]是一种体系架构框架。
<!--GB/T 22032:征求意见稿# 示例2：开放分布式处理参考模型是架构框架。-->
[来源：ISO/IEC/IEEE 42010:2011]

### 3.1.12 业务过程 business process
<!--ISO/IEC/IEEE 12207:2017# 新增术语-->

### 3.1.13 操作概念 concept of operations
<!--ISO/IEC/IEEE 12207:2017# 新增术语><GB/T 22032:征求意见稿# 4.1.11 关注 concern-->
以口头和/或图形方式概括性地说明组织对某一项或多项操作的假设或意图。
<!--GB/T 22032:征求意见稿# 文字与/或图形化表述，大致描述一个组织关于一项或一系列操作的假设或意图。-->
注1：
操作概念经常体现在长期战略计划和年度运营计划中。
在后一种情况下，计划中的操作概念包括一系列将要同时或相继进行的关联操作。
这个概念的设计是为了给出组织运营的总体情况。
另请参见操作性概念（3.1.28）。
<!--GB/T 22032:征求意见稿# 注1：操作观念体现在长期战略规划和年度操作规划中。在后者，计划中的操作概念涵盖可同时或连续执行的一系列相关操作。本概念用于描述组织操作的整体情况。也详见4.1.25。-->
注2：
它提供了操作空间、系统功能、接口和操作环境边界的基础。
<!--GB/T 22032:征求意见稿# 注2：提供确定运行空间、系统能力、界面和操作环境边界的基础。-->
[来源：ANSI/AIAA G-043A-2012e]

### 3.1.14 关注 concern
<!--ISO/IEC/IEEE 12207:2017# 新增术语><GB/T 22032:征求意见稿# 4.1.12 关注 concern-->

<!--GB/T 22032:征求意见稿# <系统>对与一个或多个利益相关方有关的系统感兴趣。-->
注：
一个关注点是关注在环境中影响系统的任何因素，包括可发展的、技术性的、业务性的、可操作性的、有组织性的、政治性的、经济性的、法律性的、管理的、生态性的、社会性的影响。
<!--GB/T 22032:征求意见稿# 注：关注涉及对系统环境的各种影响，包括可发展的、技术性的、业务性的、可操作性的、有组织性的、政治性的、经济性的、法律性的、管理的、生态性的、社会性的影响。-->
[来源：ISO/IEC/IEEE 42010:2011]

### 3.1.15 配置项 configuration item
<!--GB/T 8566-2007# 3.6 配置项 configuration item><GB/T 22032:征求意见稿# 4.1.13 技术状态项目 configuration item-->

### 3.1.20 使能系统 enabling system

<!--GB/T 22032:征求意见稿# 注：每个使能系统都有自己的生存周期。当系统自主成为所关心的系统时，本标准适用于每个使能系统。-->

### 3.1.28 操作性概念 operational concept

<!--ISO/IEC/IEEE 12207:2017# 新增术语-->
通过集成或互操作以提供独特功能的一组系统，这些独特功能无法由其任何组成系统单独完成。

注：每个组成系统本身都是一个有用的系统，且有自己的管理、目标和资源，但在体系内部，组成系统可通过协调以提供体系的独特功能。

### 3.1.65  系统工程 systems engineering
<!--ISO/IEC/IEEE 12207:2017# 新增术语>、<GB/T 22032:征求意见稿# 4.1.49 系统工程 system engineering-->
管理整个技术性和管理性成就的跨学科方法，这种成就需要将利益相关方的要求、期望、约束转化为解决方法，并且解决方法需要支持整个生存过程。

为达到一个或多个过程结果而采取的必要的、建议的或被允许的行动。

### 3.1.72 验证 verification

注1:

建立正确的系统。
<!--GB/T 22032:征求意见稿# 注：验证是一系列将系统或系统元素与需要的特性相比较的活动集合，包括但不局限于特定需求、设计描述和系统本身。系统被按照要求正确地造出来了。-->
